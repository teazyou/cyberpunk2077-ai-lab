# R1 â€” F2: record selection, navmesh placement validation, hostility wiring

## Verdict
FEASIBLE, no blockers, all 4 sub-questions closed with vanilla-cited evidence. Decisive fact: vanilla's own `DynamicSpawnSystem.SpawnCallback` (`sprint/vanilla-scripts/scripts/core/systems/dynamicSpawnSystem.script:42-56`) proves a freshly-spawned NPC does **NOT** inherit hostility from record/faction alone â€” the engine explicitly calls `ChangeAttitude()` on every spawn. We must do the same; recipe below mirrors vanilla exactly.
Record enumeration (Q2 sub-part) is **VERIFIED IMPOSSIBLE** (exhaustive check, not "expected" â€” see Findings #5). PREFERRED (same-faction) selection is feasible only via a hand-curated `array<TweakDBID>` per faction; a ready-to-use starter set (8 factions, 122 IDs) was harvested from a real published pure-REDscript mod and cross-validated against an official-style vanilla tweak-dump â€” see API inventory + Precedents. FALLBACK (verbatim clone) needs zero curation, zero extra risk â€” trivial via `GetRecordID()`.
Navmesh placement (Q3): single synchronous call, `NavigationSystem.FindPointInSphereOnlyHumanNavmesh` / `.GetNearestNavmeshPointBelowOnlyHumanNavmesh`, both vanilla-proven, no async/callback complexity.
Codeware/CET/TweakXL check: zero dependency anywhere in this dossier â€” every recommended API is either a native `import` on a vanilla class or a plain script function reachable from pure `.reds`; nothing here required a NOT-AVAILABLE ecosystem.

## Findings

1. **Identity read is a 2-hop, fully-inherited chain.** `puppet.GetRecordID() : TweakDBID` is declared on `gamePuppetBase` (`sprint/vanilla-scripts/scripts/core/gameplay/puppet.script:13`, `public import const final function GetRecordID() : TweakDBID;`) and inherited all the way to `NPCPuppet` via: `NPCPuppet extends ScriptedPuppet` (`scripts/cyberpunk/NPC/NPCPuppet.script:119`) â†’ `ScriptedPuppet extends gamePuppet` (`scripts/cyberpunk/puppet/scriptedPuppet.script:323`) â†’ `gamePuppet extends gamePuppetBase` (`scripts/core/gameplay/puppet.script:71`) â†’ `gamePuppetBase extends TimeDilatable` (`scripts/core/gameplay/puppet.script:11`) â†’ `TimeDilatable extends GameObject` (`scripts/core/systems/timeSystem.script:38`) â†’ `GameObject extends GameEntity` (`scripts/core/entity/gameObject.script:218`). Callable directly on any `ref<ScriptedPuppet>`/`ref<NPCPuppet>` in hand.

2. **Record â†’ Affiliation is a 2-call chain, vanilla-precedented verbatim.** `TweakDBInterface.GetCharacterRecord(path: TweakDBID) : Character_Record` (`scripts/core/data/tweakDB.script:371`) â†’ `Character_Record.Affiliation() : weak<Affiliation_Record>` (`scripts/core/data/tweakDBRecords.script:3476`, class starts `:3430`) â†’ `Affiliation_Record.Type() : gamedataAffiliation` (`scripts/core/data/tweakDBRecords.script:2570`) or `.EnumName() : CName` (`:2568`, class starts `:2560`). Exact combined idiom already lives in our reference mod: `sprint/reference-aggro/r6/scripts/Enemy Aggro Improvements/GunshotReactions.reds:142-143` â€” `TweakDBInterface.GetCharacterRecord( ownerPuppet.GetRecordID() ).Affiliation()`. `NPCPuppet` also ships a convenience wrapper returning a display string: `public function GetAffiliation() : String` (`scripts/cyberpunk/NPC/NPCPuppet.script:797-804`, returns `affiliation.EnumName()` or `"Unknown"`) â€” fine for debug notify, not for programmatic faction-key comparison (use `.Type()`/`.EnumName()` directly for that).

3. **Behavioral hostility ("attitude") is a separate axis from data "Affiliation" and is what actually matters for combat.** `GameObject.GetAttitudeAgent() : AttitudeAgent` (`scripts/core/entity/gameObject.script:586`) â†’ `AttitudeAgent.GetAttitudeGroup() : CName` (`scripts/core/ai/attitudeAgent.script:21`), `.GetAttitudeTowards(other: AttitudeAgent) : EAIAttitude` (`:22`). Static convenience `GameObject.GetAttitudeTowards(first, second) : EAIAttitude` (`scripts/core/entity/gameObject.script:613`) and instance form (`:632`). `EAIAttitude` enum = `{AIA_Friendly, AIA_Neutral, AIA_Hostile}` (`scripts/core/ai/attitudeAgent.script:1-6`).

4. **Two-line getter is enough; no async, no event context needed.** All of the above are plain `import`/inline functions returning values directly â€” safe to call from a tick-loop/enumeration context (matches `GetEntitiesAroundObject` pattern already used by ScannerSuite; `GameObject.GetEntitiesAroundObject(optional range, optional searchFilter) : array<Entity>` at `scripts/core/entity/gameObject.script:936`, precedent for the cast idiom `entity as ScriptedPuppet` at `mods/enabled/r6-scripts/custom-scanner-suite/ScannerSuite.reds:1081,2217`).

5. **Record enumeration is VERIFIED IMPOSSIBLE, not just "expected."** Exhaustively grepped `scripts/core/data/tweakDB.script` â€” every single one of its ~1000+ `public import static function Get*Record(path: TweakDBID) : *_Record;` accessors (sampled range `tweakDB.script:55-134` shown, pattern holds throughout the file) requires an **already-known** `TweakDBID` argument. There is no `GetRecords(className)`, no `GetAllCharacterRecords()`, no query/filter/iterate-by-type entry point anywhere in the interface. The only list-type accessors found (`GetCharacterRecordsCount()`/`GetCharacterRecordsItem(i)`, e.g. on `RandomPassengerEntry_Record` near `tweakDBRecords.script:7346-7348`) enumerate a **specific already-known record's own baked-in array**, not a global query â€” you must already hold that record's ID too. Pure REDscript cannot discover "all Character_Records with Affiliation=X" at runtime. Confirmed dead end, do not revisit.

6. **Vanilla itself uses "curated pool + weighted random pick," never enumeration â€” validates the PREFERRED approach's shape.** `PreventionSystem.GetCharacterRecordFromPool` (`scripts/core/systems/preventionSystem.script:647`, caller `GetCharacterRecordsFromPool` at `:627`) weight-randomly picks one `Character_Record` out of a `ref<array<weak<PreventionUnitPoolData_Record>>>` pool via `RandRangeF(0.0, weightSum)` + accumulator walk. This pool is police-only (`m_failsafePoliceRecordT1/2/3` at `preventionSystem.script:194-196`, e.g. `T"Character.prevention_unit_tier1"`) â€” not reusable for us since police are excluded by the F1 eligibility filter â€” but it proves "author a fixed candidate array, weighted/uniform-random-pick one" is the idiomatic, engine-native shape for this exact problem.

10. **Fallback (verbatim clone) needs nothing beyond Finding #1.** `sourceRecordID = source.GetRecordID()` reused directly as the spawn's record â€” zero curation, zero extra API surface, 100% vanilla-verified already. Always available as a safety net regardless of what the planner decides for the PREFERRED path.

11. **Navmesh: `NavigationSystem` exposes multiple synchronous (non-callback) placement/validation queries â€” verified class, full signature list.** `GameInstance.GetNavigationSystem(self: GameInstance) : NavigationSystem` (`scripts/core/systems/gameInstance.script:29`). Class `NavigationSystem extends IScriptable` (`scripts/core/systems/navigationSystem.script:53-109`); every query below is either `public import` (native, synchronous â€” returns directly, no delegate/callback) or a thin script wrapper over one:
   - `FindPointInSphereOnlyHumanNavmesh(center: Vector4, radius: Float, agentSize: NavGenAgentSize, heightDetail: Bool) : NavigationFindPointResult` (`:57`) â€” returns `{status: worldNavigationRequestStatus, point: Vector4}` struct (`:33-36`). **Point-in-sphere query**, single call, direct status check. This is the primary answer to "validated position near the source."
   - `GetNearestNavmeshPointBelowOnlyHumanNavmesh(origin: Vector4, querySphereRadius: Float, numberOfSpheres: Int32) : Vector4` (`:66-86`) â€” script-level (not native import): loops up to `numberOfSpheres` times, each iteration calls `FindPointInSphereOnlyHumanNavmesh` and on failure drops `origin.Z -= querySphereRadius` and retries (ground-snap-downward search). Returns `Vector4.Zero` on total failure â€” check via `Vector4.IsZero(result)`.
   - `IsOnGround(const target: GameObject, optional queryLength: Float) : Bool` (`:88-107`) â€” different tool: geometry/physics raycast (`GeometryDescriptionQuery`, 0.2m default length) checking whether an **existing** entity is currently standing on solid ground; not a placement-search tool, used for post-spawn/runtime footing checks (e.g. drones).
   - `FindNavmeshPointAwayFromReferencePoint(pos, refPos, distance, agentSize, out destination, optional distanceTolerance, optional angleTolerance) : Bool` (`:61`) â€” finds a point at ~`distance` from `pos` while moving away from `refPos`; shape fits "flee from threat," not "spawn near source," but usable if the planner wants the clone offset away from the player specifically.
   - `IsNavmeshStreamedInLocation(origin, findPointTolerance, optional allowedMask, optional blockedMask) : Bool` (`:62`) â€” optional pre-check that navmesh data is actually streamed in at the area (should be true near an active, visible source enemy).
   - `CalculatePathOnlyHumanNavmesh(startPoint, endPoint, agentSize, findPointTolerance, optional costModCircle) : NavigationPath` (`:55`) â€” pathfinding, not placement; could validate reachability from source to candidate point as an extra safety check.
   - `FindPursuitPoint`/`FindPursuitPointRange`/`FindPursuitPointsRange` (`:58-60`) â€” player-pursuit-shaped (needs `playerPos`+`dir`), over-fitted for our use case, not recommended.

12. **All navmesh queries confirmed synchronous via real call sites (no callback pattern anywhere).** `aiStimReactionCondition.script:390` calls `FindNavmeshPointAwayFromReferencePoint` inline inside a `Check()` and immediately branches on the bool return. `aiDeathReactionTask.script:292` and `NPCPuppet.script:4017` call `IsOnGround` inline inside an `if`. `aoeArea.script:276,287` call `CalculatePathOnlyHumanNavmesh` inline in a loop. `deviceBase.script:3244,3291` call `GetNearestNavmeshPointBelowOnlyHumanNavmesh(position, 1.0, 5)` inline â€” **real vanilla parameter values**: 1.0m query-sphere radius, 5 retry spheres.

13. **Hostility does NOT auto-inherit â€” vanilla proves it by explicitly wiring it on every spawn.** `DynamicSpawnSystem.SpawnCallback(spawnedObject: GameObject)` (`scripts/core/systems/dynamicSpawnSystem.script:42-56`): `if( spawnedObject.IsPuppet() ) { ChangeAttitude( spawnedObject, player, EAIAttitude.AIA_Hostile ); }` (call at `:55`). `ChangeAttitude` (`:75-86`): `attitudeOwner = owner.GetAttitudeAgent(); attitudeTarget = target.GetAttitudeAgent(); attitudeOwner.SetAttitudeGroup( 'hostile' ); attitudeOwner.SetAttitudeTowards( attitudeTarget, desiredAttitude );` (lines `:83-90`). This is decisive: if hostility were automatic from record/Affiliation alone, the engine's own spawn system would not need this.

14. **Full `AttitudeAgent` mutator inventory** (`scripts/core/ai/attitudeAgent.script:18-26`): `SetAttitudeGroup(group: CName)` (`:23`), `SetAttitudeGroupUnsavable(group: CName)` (`:24`), `SetAttitudeTowards(agent: AttitudeAgent, attitude: EAIAttitude)` (`:25`), `SetAttitudeTowardsAgentGroup(targetAgent, ownerAgent, attitude)` (`:26`).

15. **Better-fidelity variant: copy the SOURCE's attitude group onto the clone instead of hardcoding `'hostile'`.** Precedent: `aiRole.script` private `ChangeAttitude` helper (follower role) â€” `ownAttitudeAgent.SetAttitudeGroup( targetAttitudeAgent.GetAttitudeGroup() )` (`scripts/cyberpunk/ai/roles/aiRole.script:315,327`). Same idiom at `scripts/core/gameplay/effectors/quickhackEffectors.script:517`. Recommended: `clone.GetAttitudeAgent().SetAttitudeGroup( source.GetAttitudeAgent().GetAttitudeGroup() )` then explicit `SetAttitudeTowards(player's agent, AIA_Hostile)` â€” matches the source's real in-game group (which may not literally be `'hostile'` for every gang) while still guaranteeing player-hostility.

16. **"Fights immediately" (not just flagged hostile) needs a combat-threat injection, not just attitude â€” vanilla ships exactly this for DynamicSpawnSystem spawns.** `DynamicSpawnSystem.SpawnRequestFinished` (`dynamicSpawnSystem.script:18-40`) builds and sends an `AIInjectCombatThreatCommand` for every spawned puppet:

17. **Alternate/complementary hostility trigger: stim broadcast, all APIs independently vanilla-verified.** `StimBroadcasterComponent.SendStimDirectly(sender: weak<GameObject>, gdStimType: gamedataStimType, target: GameObject, optional investigateData: stimInvestigateData, optional delay: Float)` â€” static (`scripts/core/components/stimBroadcasterComponent.script:225`, class `:99`). `GameObject.GetStimBroadcasterComponent() : StimBroadcasterComponent` (`scripts/core/entity/gameObject.script:3075`). `gamedataStimType.CombatHit` confirmed in enum (`scripts/core/data/tweakDBEnums.script:2938`). `GameObject.HasTag(tag: CName) : Bool` (`scripts/core/entity/gameObject.script:271`) for the tag-gate. ## Dead ends
- **TweakDB record enumeration by class/query â€” does not exist.** Exhaustively checked `tweakDB.script`'s entire import list; every `Get*Record` needs a known ID. No amount of further grepping will find one; do not re-attempt.
- **Local `sprint/vanilla-scripts` for generic per-faction archetype TweakDBIDs â€” does not contain them.** Decompiled script *logic* only ever references named/quest/boss characters by literal TweakDBID; generic grunt archetypes live in TweakDB *data* (tweak/archive files), outside this repo's scope. Required going external (GitHub code search via `gh search code`, not plain WebSearch â€” WebSearch alone returned only vague faction-name hits, `gh search code` found the exact source files in one shot).
- **`AISquadHelper` â€” no "join a squad" setter.** Only leave/pull-sync/enumerate/alert-existing-members. Squad membership is data/proximity-authored, not a single callable. Don't design around it; use `AIInjectCombatThreatCommand` instead (Finding #16/18).
- **WolvenKit-style TweakDB GUI browsing â€” not attempted, correctly out of scope.** Would require extracting/browsing the live game archive with a GUI tool not available in this environment; `gh search code` against public tweak-dump repos was the effective substitute and succeeded.
